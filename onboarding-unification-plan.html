Onboarding Unification — Implementation Plan

Kinpath Family · March 29, 2026 · Consolidating web & mobile into one consistent onboarding flow

1. Target Unified Flow

After this work, every new user — regardless of platform (web or mobile) or auth method (email, Google, Apple) — will follow the same logical sequence:

Scroll horizontally if needed →

1. Register

Email + password, Google, or Apple. OAuth available on both login and register pages.

→

2. Email Confirm

If required by Supabase (email signups). OAuth skips this.

→

3. Child Profile

Name, born/expecting, DOB or due date. Required.

→

4. Preferences

Birth, feeding, vaccines, lifestyle, topics. Skippable steps.

→

5. Paywall

Stripe (web) or RevenueCat (mobile). Soft prompt — skippable.

→

6. Partner Invite

Optional invite. Skip always available.

→

7. Complete

Set onboarding_complete=true only here. Redirect to dashboard/tabs.

Key principle: onboarding_complete is only set to true at the very end of the flow — after child profile, preferences, paywall, and partner invite have all been presented. No step sets it early.

2. Implementation Phases

1
Add onboarding step tracking to the database
SHARED   ~0.5 day

Replace the boolean onboarding_complete with a step tracker so the user can resume mid-flow if they close the app or browser. The boolean stays as a computed convenience, but the step field becomes the source of truth for routing.

Migration

-- New migration: add onboarding_step column
ALTER TABLE public.users
  ADD COLUMN onboarding_step text NOT NULL DEFAULT 'child'
  CHECK (onboarding_step IN (
    'child', 'preferences', 'paywall', 'partner_invite', 'complete'
  ));

-- Backfill: users who already finished stay finished
UPDATE public.users
  SET onboarding_step = 'complete'
  WHERE onboarding_complete = true;

-- Users who never completed start from the beginning
UPDATE public.users
  SET onboarding_step = 'child'
  WHERE onboarding_complete = false;
  • 📄
    Create new SQL migration file in packages/supabase/migrations/
  • 📄
    Update the User type in packages/shared/src/types/user.ts to include onboarding_step
  • 📄
    Add OnboardingStep type to packages/shared: "child" | "preferences" | "paywall" | "partner_invite" | "complete"
  • 📄
    Keep onboarding_complete as a computed check — onboarding_step = 'complete' means onboarding_complete = true. Update the handle_new_user() trigger to set onboarding_step = 'child' for new users.
Important: Keep the existing onboarding_complete column in sync via a DB trigger or by updating both fields together. This avoids breaking any existing queries that check the boolean.

    packages/mobile/app/_layout.tsx (lines 23, 78-86)
// BEFORE:

  ? new Date(userCreatedAt) > POST_AUTH_LAUNCH
  : false;

  router.replace("/(post-auth)/paywall");
}

// AFTER:
if (onboardingStep && onboardingStep !== 'complete') {
  // Resume from wherever the user left off
  const screenMap = {
    child: 'child-profile',
    preferences: 'birth-preference',
    paywall: 'paywall',
    partner_invite: 'partner-invite',

  router.replace(`/(post-auth)/${screenMap[onboardingStep]}`);
}

Edge cases to verify

  • ⚠️
    Stripe checkout cancellation: User clicks "Start Free Trial" on web → goes to Stripe → clicks browser back. Should return to onboarding at the paywall step, not a broken state.
  • ⚠️
    RevenueCat timeout: waitForIdentification() has a 5-second timeout. If it expires, the paywall should still be skippable.
  • ⚠️
    Network failure during DB save: If child insert or preference upsert fails, user should see an error and be able to retry — not advance silently.
  • ⚠️
    Dual-platform user: User starts onboarding on mobile, abandons at "topics" step, then logs in on web. Web should read onboarding_step from DB and resume at the correct point.

4. Dependency Order

Phases must be executed in order because of data dependencies:

Phase 1: DB migration → Phase 2: Web and Phase 3: Mobile (parallel) → Phase 4: Guards → Phase 5: Testing
Total estimated effort: 7–8.5 days. Phase 1 unblocks everything. Phases 2 and 3 can be developed in parallel by separate developers (or sequentially by one). Phase 4 is quick cleanup. Phase 5 covers the full test matrix.

5. Risks & Mitigations

RiskImpactMitigation
Middleware DB query on every request slows page loads Latency on all authenticated web pages Cache onboarding_complete in a session cookie after it becomes true. Only query the DB when the cookie is missing or false.
Stripe redirect loses onboarding context User ends up on wrong page after payment Use return_url param on checkout + ?step=partner_invite query string. Read onboarding_step from DB on mount as backup.
Existing users with onboarding_complete=false forced through full flow Friction for long-time users who never hit the old onboarding The migration backfills onboarding_step='child' for these users, which is correct — they genuinely have no child profile. A banner could explain why they're seeing onboarding.
RevenueCat and Stripe paywall screens look/feel different Inconsistent brand experience The web paywall is custom-built (Stripe redirect), so match Kinpath branding. RevenueCat allows paywall template customization in their dashboard.
Cross-platform resume (start on mobile, finish on web) Could show duplicate steps or skip steps onboarding_step is the single source of truth in the DB. Both platforms read it on mount and jump to the right step.
